### 1a: Structural Survey

Harnessiq is a Python 3.11+ SDK organized around manifest-backed agents under `harnessiq/agents/`, provider adapters under `harnessiq/providers/`, tool factories under `harnessiq/tools/`, shared durable-memory/config models under `harnessiq/shared/`, and argparse CLI entrypoints under `harnessiq/cli/`. The live runtime source of truth is the `harnessiq/` package; generated repo-reference artifacts live under `artifacts/` and are regenerated by `python scripts/sync_repo_docs.py`.

The concrete harnesses in scope for this task are the agent packages under `harnessiq/agents/`: `exa_outreach`, `instagram`, `knowt`, `leads`, `linkedin`, `prospecting`, and `research_sweep`, plus abstract/shared agent layers in `apollo`, `email`, `exa`, `instantly`, `outreach`, `provider_base`, and `base`. Concrete agents subclass `BaseAgent` and typically keep orchestration logic in `agent.py` while reading durable memory from corresponding store classes in `harnessiq/shared/*`. Many agent modules currently also define module-local helper functions at the bottom of `agent.py` for instance payload building, optional file reads, repo-root resolution, tool-definition construction, and timestamp generation.

The Instagram harness is split across `harnessiq/agents/instagram/agent.py`, `harnessiq/shared/instagram.py`, `harnessiq/tools/instagram.py`, `harnessiq/tools/instagram/operations.py`, and CLI surfaces in `harnessiq/cli/instagram/commands.py` plus `harnessiq/cli/adapters/instagram.py`. Tests are `unittest`-based in `tests/`, with targeted Instagram coverage in `tests/test_instagram_agent.py` and `tests/test_instagram_cli.py`. The repository does not define a dedicated linter or type checker in `pyproject.toml`; the existing workflow relies on `python -m unittest` and repo-specific generation checks such as docs sync.

Conventions visible across the codebase:
- Agents orchestrate model loops and durable-memory interaction; tools execute deterministic actions; providers wrap external systems.
- Runtime and custom parameters are persisted in JSON files and injected back into model parameter sections.
- Agent modules often include small private helper functions in the same file rather than a sibling helper module.
- Multiple agent modules repeat identical or near-identical helpers such as optional text reads, repo-root discovery, memory-path resolution, and UTC timestamp formatting.
