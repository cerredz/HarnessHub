### 1a: Structural Survey

#### Top-level architecture

- `harnessiq/` is the only live runtime package. `build/`, `src/`, and egg metadata are generated or residue directories.
- `harnessiq/agents/` owns the runtime loop and concrete harnesses. `harnessiq/agents/base/agent.py` and `harnessiq/agents/base/agent_helpers.py` are the core orchestration seam for prompt assembly, tool visibility, tool execution, transcript recording, resets, and formalization-layer participation.
- `harnessiq/interfaces/` is the newer SDK-facing contract surface. `harnessiq/interfaces/formalization/` already contains split abstract bases for formalization concerns (`base.py`, `contract.py`, `artifact.py`, `stage.py`, `state.py`, `hook_layer.py`, `role.py`, `tool_contribution.py`).
- `harnessiq/shared/` holds dependency-light DTOs and constants. `harnessiq/shared/formalization.py` is the shared vocabulary for formalization metadata. `harnessiq/shared/tools.py` defines the canonical tool constants and the `ToolDefinition`, `RegisteredTool`, `ToolCall`, and `ToolResult` runtime models.
- `harnessiq/formalization/` is an older runtime package that still contains a monolithic formalization base plus the only current concrete runtime layer family (`stages/`).
- `tests/` is substantial and pytest/unittest-first. Runtime behavior is primarily covered by `tests/test_agents_base.py`, `tests/test_formalization_stages.py`, and `tests/test_interfaces.py`.
- `artifacts/file_index.md` and companion repo docs are generated by `python scripts/sync_repo_docs.py`; they should not be hand-edited.

#### Relevant runtime data flow

1. `BaseAgent` resolves and stores `formalization_layers` in `harnessiq/agents/base/agent.py`.
2. `BaseAgentHelpersMixin` applies layer hooks in this order:
   - `on_agent_prepare`
   - `augment_system_prompt`
   - `get_parameter_sections`
   - `filter_tool_keys`
   - `on_tool_result`
   - `on_pre_reset` / `on_post_reset`
3. Tool calls are recorded into the transcript before execution, but `on_tool_result` currently receives only `ToolResult(tool_key, output)` and not the original arguments.
4. Formalization-contributed tools are overlaid onto the executor by `_collect_formalization_tools()`, which currently expects `RegisteredTool` instances at runtime.

#### Existing conventions

- Interface modules are one concept per file, with strong top-of-file prose docstrings and typed abstract methods.
- Shared formalization records live in `harnessiq/shared/formalization.py`; interface behavior is built from those records.
- Public exports are maintained in three places:
  - `harnessiq/interfaces/formalization/__init__.py`
  - `harnessiq/interfaces/formalization.py`
  - `harnessiq/interfaces/__init__.py`
- Tests validate both self-documentation and runtime behavior. The codebase expects deterministic, inspectable rule records rather than opaque prompt-only conventions.

#### Inconsistencies already present

- There are two formalization stacks:
  - `harnessiq/interfaces/formalization/*` is the newer decomposed contract package.
  - `harnessiq/formalization/base.py` is the older monolith still consumed by `BaseAgent`.
- `harnessiq/interfaces/formalization.py` is labeled as a compatibility export but currently re-exports from `harnessiq.formalization.base`, not from `harnessiq/interfaces/formalization/`.
- `BaseToolContributionLayer` in the interface package is typed to return `ToolDefinition`, while the runtime overlay in `BaseAgentHelpersMixin` actually requires `RegisteredTool`.

### 1b: Task Cross-Reference

#### Requested change mapped onto the repo

- New behavior formalization package:
  - Net-new package under `harnessiq/interfaces/formalization/behaviors/`
  - Net-new category subpackages for tool, pace, reasoning, quality, recovery, safety, and communication
  - Net-new `__init__.py` re-exports
- Shared behavior abstractions:
  - `BehaviorConstraint` and `BaseBehaviorLayer` belong under the new behavior package
  - New typed abstract bases should follow the existing interface-package pattern used by `contract.py`, `artifact.py`, and `stage.py`
- Runtime integration:
  - `harnessiq/agents/base/agent.py` needs the new `behaviors=` sugar and ordering semantics
  - `harnessiq/agents/base/agent_helpers.py` needs richer hook plumbing because the requested design depends on argument-aware enforcement
- Export surface:
  - `harnessiq/interfaces/formalization/__init__.py`
  - `harnessiq/interfaces/formalization.py`
  - `harnessiq/interfaces/__init__.py`
  - legacy-compatible exports under `harnessiq/formalization/`
- Tests:
  - extend `tests/test_interfaces.py` for new abstract bases and exports
  - extend `tests/test_agents_base.py` for `behaviors=` integration
  - add focused behavior-layer tests
- Generated repo docs:
  - `artifacts/file_index.md` should be regenerated via `python scripts/sync_repo_docs.py` after source changes

#### What already exists that is reusable

- `BaseFormalizationLayer` in `harnessiq/interfaces/formalization/base.py` already provides the right self-documenting pattern and lifecycle hooks.
- `LayerRuleRecord` and `FormalizationDescription` in `harnessiq/shared/formalization.py` are the existing auditable rule vocabulary.
- `is_tool_allowed()` in `harnessiq/tools/hooks/defaults.py` already implements the tool-pattern matching style used elsewhere.
- Control and reasoning tool constants already exist in `harnessiq/shared/tools.py`.

#### What is missing

- No behavior-layer package exists under `harnessiq/interfaces/formalization/`.
- `BaseAgent` has no `behaviors=` parameter and still imports legacy formalization types from `harnessiq.formalization`.
- No existing formalization hook can inspect tool arguments before execution from inside a layer.
- No existing behavior-specific helper tools such as `behavior.confirm_action` or `behavior.flag_scope_violation` exist.

#### Blast radius

- Public API surface under `harnessiq.interfaces` and legacy-compatible `harnessiq.formalization`
- Base runtime constructor and layer ordering behavior
- Formalization-contributed tool expectations
- Test snapshots around exports and runtime integration
- Generated architecture artifacts

### 1c: Assumption & Risk Inventory

1. Assumption: behavior layers should live under `harnessiq/interfaces/formalization/behaviors/` and be treated as part of the newer interface package.
   Risk: without aligning `BaseAgent`, the new layers would exist but not be first-class in the runtime API.

2. Assumption: the task intends a real runtime implementation, not interface-only abstract contracts.
   Risk: the design doc mixes interface classes, concrete runtime enforcement, and `BaseAgent` constructor changes. Implementing only the interface package would leave the design half-finished.

3. Assumption: the runtime may be extended so behavior layers can inspect tool-call arguments or pre-tool execution context.
   Risk: several proposed behaviors cannot be implemented faithfully with the current hooks:
   - `RetryStrategyBehavior`
   - `StuckDetectionBehavior`
   - `ScopeGuardBehavior`
   - any safety/recovery rule that depends on exact arguments before execution

4. Assumption: only `behaviors=` should be added to `BaseAgent` in this task.
   Risk: the design doc references artifact ordering, but the requested implementation scope excludes adjacent artifact constructor sugar.

5. Assumption: compatibility exports matter.
   Risk: today’s public import graph is inconsistent. Adding behaviors only to `interfaces/formalization/__init__.py` would leave the legacy and top-level compatibility surfaces incomplete.

6. Assumption: the full concrete matrix should be implemented now.
   Risk: this is a broad change that will require careful ticketing and a stronger runtime hook contract, not just new class shells.
