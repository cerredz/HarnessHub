### 1a: Structural Survey

Top-level architecture:
- `harnessiq/` is the authoritative runtime package. The repository is a Python 3.11+ SDK/CLI with setuptools packaging from `pyproject.toml`.
- `harnessiq/agents/`, `harnessiq/providers/`, `harnessiq/tools/`, `harnessiq/cli/`, `harnessiq/shared/`, `harnessiq/utils/`, and `harnessiq/toolset/` reflect a deliberate separation of orchestration, external integrations, deterministic tool surfaces, CLI entrypoints, shared manifests/types, and cross-cutting utilities.
- `scripts/` contains repository-maintenance tooling. `scripts/sync_repo_docs.py` is the relevant generator for README and artifact docs.
- `artifacts/` stores generated repository reference artifacts. At baseline it contains `file_index.md`, `commands.md`, and `live_inventory.json`.
- `docs/` contains focused hand-authored reference material.
- `tests/` provides Python test coverage, using `unittest` style tests executed via `pytest`.
- `memory/` stores prior task artifacts plus runtime state and is part of established repository workflow.

Technology and tooling:
- Python packaging via setuptools; CLI exposed as `harnessiq`.
- No linter or type checker is configured in `pyproject.toml`.
- Tests are organized under `tests/` and use regression-style coverage around runtime, CLI, providers, and repo tooling.

Relevant data flow:
1. `scripts/sync_repo_docs.py` introspects the live repository and runtime package.
2. It builds an in-memory inventory via `build_inventory()`.
3. It renders generated outputs through `expected_outputs()`.
4. It writes and/or checks those outputs, including `README.md` and files under `artifacts/`.
5. `tests/test_docs_sync.py` verifies generated docs remain aligned with the live source tree.

Conventions observed:
- Generated repository docs are treated as committed artifacts and should be regenerated from code rather than edited manually.
- README content that documents generated artifacts is sourced from constants inside `scripts/sync_repo_docs.py`.
- The codebase prefers narrowly scoped regression tests close to the changed behavior.
- Durable memory and generated repo metadata are treated as first-class operational concerns.

Inconsistencies and notable conditions:
- The working tree is already dirty on `main`, including many unrelated modified and untracked files.
- `build/`, `src/`, `.egg-info`, and caches are present as residue/generated directories even though `harnessiq/` is the stated source of truth.
- The repository currently commits a large machine-readable generated artifact (`artifacts/live_inventory.json`) even though the human-facing generated docs are `README.md`, `artifacts/file_index.md`, and `artifacts/commands.md`.

### 1b: Task Cross-Reference

User task: remove `artifacts/live_inventory.json` from GitHub because it consumes unnecessary tokens.

Concrete codebase mapping:
- `artifacts/live_inventory.json`
  - Tracked generated artifact targeted for removal from version control.
- `scripts/sync_repo_docs.py`
  - Declares the generated repo-doc links shown in `README.md`.
  - Generates `artifacts/live_inventory.json` in `expected_outputs()`.
  - Owns the write/check path for repo-doc outputs, so this is the canonical place to stop regenerating the file and to define any cleanup behavior for the removed artifact.
- `README.md`
  - Generated by `scripts/sync_repo_docs.py` and currently advertises `artifacts/live_inventory.json` in the Repo Docs section.
- `tests/test_docs_sync.py`
  - Existing regression coverage for the docs sync generator. This is the correct place to add/adjust tests so the removed artifact does not silently re-enter the generated output set later.

Behavior to preserve:
- `scripts/sync_repo_docs.py --check` must still validate generated repo docs.
- `README.md`, `artifacts/file_index.md`, and `artifacts/commands.md` must remain generated from live source.
- Existing inventory-driven tests that rely on `build_inventory()` should continue to pass.

Missing work implied by the task:
- Removing the file from git alone is insufficient because the generator would recreate it.
- README references must be removed because the artifact is currently documented as part of the public repo-doc surface.
- Regression coverage should enforce the artifact’s removal from the generated output contract.

Blast radius:
- Limited to the repo-doc generation pipeline and one generated artifact.
- No runtime SDK behavior, provider logic, or CLI behavior should change.

### 1c: Assumption & Risk Inventory

Assumptions:
- “Remove from GitHub” means stop tracking and stop generating `artifacts/live_inventory.json`, not merely add it to `.gitignore`.
- The artifact is not consumed by runtime package code; current repository references indicate it is only part of the repo-doc generation surface.
- The desired outcome is to keep the human-readable artifacts (`README.md`, `artifacts/file_index.md`, `artifacts/commands.md`) and drop the machine-readable inventory artifact.

Risks:
- Because the working tree is already dirty on `main`, any workflow that mutates the root checkout risks colliding with unrelated user changes.
- If the generator stops emitting the file but does not actively clean it up, stale local copies could persist and be recommitted later.
- If README generation is not updated alongside the output set, generated docs will drift or continue advertising a removed file.
- If tests only check live inventory-building behavior and not the emitted output contract, the artifact could accidentally reappear in the future.

Resolution posture:
- Keep implementation isolated to the repo-doc generation workflow.
- Use a clean worktree branched from fetched `origin/main` rather than modifying the dirty root checkout.
- Add regression coverage around the generator’s declared outputs and legacy artifact cleanup semantics.
